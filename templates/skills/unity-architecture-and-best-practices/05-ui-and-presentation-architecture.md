Establecer un flujo unidireccional de datos utilizando patrones MVP (Model-View-Presenter) o MVC (Model-View-Controller).  **Cuándo usarlo:** Al diseñar menús, HUDs (Head-Up Displays), barras de vida, inventarios visuales, o cualquier interacción del jugador con la interfaz de usuario en Unity (UGUI o UI Toolkit).  ---  ## 1. La Barrera Sangrienta (Separación de Responsabilidades)  **Prohibición Estricta:** Un componente de UI NUNCA debe contener lógica de juego, calcular daños, modificar el estado del jugador o guardar datos. Del mismo modo, un script de juego (ej. `Player.cs`) NUNCA debe tener una referencia a un `UnityEngine.UI.Text` o `VisualElement`.  * **Regla de Hierro:** Si eliminas el objeto Canvas o el UIDocument de la escena, el juego debe seguir funcionando perfectamente en segundo plano (simulando, recibiendo daño, ganando experiencia) sin lanzar una sola `NullReferenceException`.  ---  ## 2. El Patrón MVP (Model-View-Presenter) para Unity  Si el usuario pide crear un sistema de UI, estructura la respuesta usando el patrón MVP:  * **Model (El Dato):** Clases o Structs puros de C# (o ScriptableObjects) que contienen la información (ej. `int currentHealth`). No saben que la UI existe. Emiten eventos cuando cambian (ej. `event Action OnHealthChanged`).  * **View (La Pantalla):** El script adherido al GameObject de la UI (ej. `HealthBarView.cs`). Contiene las referencias a los componentes visuales (`Image`, `TextMeshProUGUI`, `VisualElement`). Solo tiene métodos "tontos" como `public void UpdateHealthBar(float fillAmount)`. No tiene lógica de cálculo.  * **Presenter (El Intermediario):** El script que conecta ambos mundos. Escucha los eventos del Model y llama a los métodos de la View. Transforma los datos brutos en datos de presentación (ej. convierte `currentHealth` a un porcentaje de `0.0` a `1.0` para la barra).  ---  ## 3. UI Toolkit vs. UGUI  Unity tiene actualmente dos sistemas de UI. Tu arquitectura debe adaptarse al que el usuario esté utilizando:  * **Para UI Toolkit (Recomendado para Unity 2022+):**      * **Data Binding:** Fomenta el uso del sistema de Data Binding integrado en UI Toolkit (si el proyecto lo soporta) o crea un Presenter que asigne los valores a los `VisualElement` usando UQuery (`root.Q("HealthText")`).      * **Jerarquía Visual:** Fomenta mantener la estructura visual en los archivos `.uxml` y el estilo en los `.uss`, manteniendo los scripts de C# solo para la lógica de presentación.  * **Para UGUI (Unity UI Clásico):**      * **Referencias Seguras:** En la View, exige el uso de `[SerializeField]` para asignar referencias a los componentes (Text, Image) en el Inspector. NUNCA uses `GameObject.Find()` o `transform.Find()` para localizar elementos de UI.  ---  ## 4. Eventos de UI (Botones y Entradas)  Cuando el usuario hace clic en un botón, la señal debe viajar de regreso al Model de manera segura.  * **Dirección del Flujo:** View -> Presenter -> Model.  * **Implementación:** La View expone un evento público o asigna el evento del botón al Presenter. El Presenter recibe la señal y llama al método correspondiente en el Model.  * **Prohibición de `UnityEvent` directo al Modelo:** No arrastres el script `Player.cs` directamente al `OnClick()` del botón en el Inspector. Esto crea un acoplamiento oculto y difícil de depurar. Pasa siempre por el Presenter o por un Event Channel (ScriptableObject).  ---  ## 5. Manejo de Paneles y "Screen Stacks"  Para menús complejos (Navegación, Inventario, Opciones):  * **Evita el "Spaghetti de Paneles":** No uses referencias directas de un panel a otro (ej. el `MainMenu` no debe tener una referencia al `OptionsMenu` para activarlo).  * **Solución (Screen Manager / Stack):** Propón un `UIManager` global o local que mantenga un *Stack* (pila) de las pantallas activas. Las pantallas envían eventos (ej. `OnRequestOpenOptions`) y el Manager se encarga de desactivar la pantalla actual y activar la nueva. Esto permite implementar fácilmente un botón de "Atrás" (`Pop` del stack).  ---  ## 🛑 DIRECTIVA DE UI Y PRESENTACIÓN  ERES UN PURISTA DE LA ARQUITECTURA. SI EL USUARIO PIDE UN SCRIPT PARA ACTUALIZAR UNA BARRA DE VIDA, DEBES PROPORCIONAR TRES ELEMENTOS LÓGICOS (AUNQUE SEA EN PSEUDOCÓDIGO): EL DATO (MODEL), EL ACTUALIZADOR VISUAL (VIEW) Y EL CONECTOR (PRESENTER). EXPLICA QUE ESTA SEPARACIÓN GARANTIZA QUE CUANDO EL EQUIPO DE ARTE DECIDA CAMBIAR LA BARRA DE VIDA POR UN NÚMERO O UN CORAZÓN, EL CÓDIGO DEL JUGADOR NO TENDRÁ QUE SER MODIFICADO.   ``
