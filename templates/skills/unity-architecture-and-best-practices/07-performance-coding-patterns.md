El Costo de Mover Objetos (Transformaciones)  Modificar un `Transform` en Unity no es barato, ya que puede desencadenar actualizaciones en las físicas, en las jerarquías de hijos y en el renderizado.  * **Regla de la Modificación Local:** Altera `localPosition`, `localRotation` y `localScale` en lugar de sus contrapartes globales (`position`, `rotation`), a menos que sea estrictamente necesario. Las modificaciones globales obligan a Unity a recalcular matrices a través de toda la jerarquía de padres e hijos.  * **Cacheo de Referencias:** NUNCA llames a `Camera.main` en un `Update`. Esta propiedad hace un equivalente a `FindObjectWithTag` en versiones antiguas, o aunque esté optimizada en versiones recientes, sigue siendo más lenta que una referencia cacheada.    * *Acción Obligatoria:* `private Camera _mainCamera; void Awake() { _mainCamera = Camera.main; }`  ---  ## 2. APIs NonAlloc (Cero Basura en Físicas)  Las comprobaciones espaciales de físicas (Raycasts, SphereCasts) son fundamentales, pero sus versiones estándar crean arreglos (arrays) en el heap, generando GC Spikes.  * **Prohibición Estricta:** NO uses `Physics.RaycastAll` ni `Physics.OverlapSphere` en bucles regulares.  * **La Solución:** Usa las variantes **NonAlloc**.    * Exige el uso de `Physics.RaycastNonAlloc` o `Physics.OverlapSphereNonAlloc`.    * *Implementación:* Debes crear un arreglo pre-asignado de `RaycastHit` o `Collider` en la clase y pasar ese arreglo a la función.    ```csharp    private readonly Collider[] _hitResults = new Collider[10];    private void CheckAoE()    {        int hitCount = Physics.OverlapSphereNonAlloc(transform.position, _radius, _hitResults, _layerMask);        for (int i = 0; i < hitCount; i++) { /* Procesar impacto */ }    }   ``  3. Búsqueda y Comunicación Letales ----------------------------------  Unity ofrece métodos que son trampas mortales para el rendimiento si se usan en tiempo de ejecución.  *   **Find y Tags:** Prohibido usar GameObject.Find(), GameObject.FindWithTag(), o FindObjectOfType() fuera de rutinas de inicialización (Awake/Start) o código de Editor. Las referencias deben inyectarse o asignarse en el Inspector.      *   **SendMessage:** Prohibido usar SendMessage() o BroadcastMessage(). Utilizan reflexión (Reflection), son lentos, oscurecen las dependencias y fallan en silencio si se renombra un método. Usa interfaces, eventos o llamadas a métodos directos con referencias cacheadas.      *   **Comparación de Tags:** No uses if (gameObject.tag == "Player") (genera un nuevo string y aloca memoria). Usa el método optimizado: if (gameObject.CompareTag("Player")).       4. Time-Slicing (Cortando el Trabajo Pesado) --------------------------------------------  Si un sistema debe procesar miles de entidades o cálculos (ej. IA de 500 enemigos) y no estás usando el Job System (DOTS):  *   **La Solución:** No calcules todo en el mismo frame. Usa _Time-slicing_ (reparto de carga en el tiempo).      *   **Implementación:** Divide la lista en porciones (chunks). Si tienes 1000 enemigos, procesa 100 en el frame 1, 100 en el frame 2, etc. Puedes implementar esto con un índice persistente en el Update o usando una Corrutina que ejecute yield return null después de procesar X elementos.       5. Colecciones Optimizadas (Listas vs Arrays) ---------------------------------------------  La elección del contenedor de datos afecta enormemente la memoria y la velocidad.  *   **Arrays []:** Úsalos cuando el tamaño de los datos es fijo y conocido. Son la estructura de datos más rápida y "cache-friendly" en C#.      *   **Lists :** Úsalas solo cuando el tamaño de la colección varíe constantemente.          *   _Pre-dimensionamiento obligatorio:_ Si sabes que una lista va a contener aproximadamente 50 elementos, inicialízala con capacidad: List enemies = new List(50);. Esto evita redimensionamientos costosos bajo el capó cuando la lista crece.           🛑 DIRECTIVA DE RENDIMIENTO MICRO ---------------------------------  TU OBJETIVO AL ESCRIBIR LA IMPLEMENTACIÓN DE UN MÉTODO ES PREGUNTARTE: "¿ESTO SE EJECUTARÁ CADA FRAME?".
